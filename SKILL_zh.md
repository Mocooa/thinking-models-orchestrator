# 思维模型编排器

你已经掌握了所有模型知识。这个 skill 告诉你**如何选择、组合、运用这些模型**，产出真正让用户惊喜和受益的分析。

## 核心原则

- **编排而非说教** —— 用户要的是洞察，不是教科书
- **先质疑再分析** —— 分析之前先审视问题本身
- **多镜头是必须的** —— 单一学科的分析是不充分的
- **矛盾是金矿** —— 永远不要掩盖矛盾；它们揭示核心张力
- **深度胜于广度** —— 2-3 个模型深入使用，胜过 6 个模型浅尝辄止

---

## 第 0 步：质疑问题

在任何分析之前，从三个维度审视用户的问题：

### 隐含假设
问题预设了什么？这个预设成立吗？
- "如何提高转化率？" → 预设了转化率是瓶颈。真的是吗？
- "该不该扩张？" → 预设了扩张和不扩张是仅有的两个选项。

### 问题层级
用户问的是症状还是根因？
- 症状级："收入下降了怎么办？"
- 根因级："为什么客户不再复购？"
- 如果是症状级，先引导到根因。

### 框架锁定
用户是否锁定在某个视角里？
- "从营销角度分析..." → 也许产品本身才是问题。
- 如果检测到框架锁定，主动提供替代框架。

**行动**：如果发现了重要的假设/层级/锁定问题，向用户指出，并问："你想先探讨这个，还是继续原来的问题？"如果问题本身没有明显问题，直接跳到第 1 步。

---

## 第 1 步：快速匹配

检查问题是否符合已知的特征信号。如果是，直接使用对应的模型组合。

| 特征信号 | 模型组合 | 领域镜头 |
|----------|---------|---------|
| 二选一/多选一，有明确选项 | Pros/Cons + Inversion | 机会成本、二阶效应 |
| 已知结果不好，需找原因 | 5 Whys + MECE | 激励机制、反馈循环、幸存者偏差 |
| 需要新想法或改进方案 | SCAMPER + First Principles | 生态位、杠杆、类比推理 |
| 面向未来的不确定性 | Scenario Planning + Bayesian | 引爆点、路径依赖、黑天鹅 |
| 需要说服/汇报/pitch | Golden Circle + Minto | 社会认同、损失厌恶、锚定 |
| 防御性——预防失败/风险 | Pre-mortem + Inversion | 确认偏误、帕金森定律 |
| 进入陌生领域/市场 | JTBD + MECE | 比较优势、网络效应、临界质量 |
| 长周期战略规划 | ToT + Second-Order | 飞轮、护城河、创新者困境 |
| 系统性衰退（用户/收入/质量） | 5 Whys + Bottleneck | 反馈循环、摩擦力、峰终定律 |
| 定价/估值 | Trade-off + Bayesian | 供需、锚定、信号传递、价格弹性 |
| 完全没头绪，不知从何入手 | Cynefin 先分类 | — |

**匹配成功** → 使用该组合，根据需要补充额外镜头，然后跳到第 4 步。
**无匹配** → 进入第 2 步。

---

## 第 2 步：分解复合问题

很多问题是复合的——包含需要不同模型的子问题。

**分解信号**：问题同时包含"为什么"和"怎么办"，或跨越多个阶段（理解 → 创造 → 评估 → 沟通）。

**示例**："用户流失严重，怎么办？"
1. **为什么？**（分析）→ 5 Whys + MECE 找根因
2. **有什么方案？**（创意）→ SCAMPER 生成解决方案
3. **选哪个？**（决策）→ Pros/Cons + Inversion 评估
4. **如何汇报？**（沟通）→ Minto Pyramid 结构化表达

将每个子问题独立送入第 3 步。使用 CGEC（见附录 A）或其他元框架来编排各阶段。

---

## 第 3 步：三维诊断

对每个（子）问题，确定三件事：

### A. 任务类型 → 选择实用模型

| 任务类型 | 首选 | 备选 |
|----------|------|------|
| 理解/推理 | CoT | ToT, 5 Whys, Socratic, Self-Consistency |
| 创造/创新 | SCAMPER | First Principles, Lateral Thinking, Analogical |
| 决策/选择 | Pros/Cons Matrix | Inversion, Second-Order, Pre-mortem, Trade-off |
| 分析/诊断 | MECE + 5 Whys | Pareto, Bottleneck, Bayesian |
| 沟通/说服 | Golden Circle | Minto Pyramid, AIDA, Feynman |
| 预测/预判 | Scenario Planning | JTBD, Black Swan, Bayesian |

### B. 领域镜头 → 强制排序（选 2-3 个，不要全选）

从以下 6 个领域中，选出与问题**最相关的 2-3 个**：

- **心理学**：行为、动机、认知偏误
- **经济学**：资源配置、市场机制、博弈论
- **生物学/战略**：竞争、演化、生态位
- **系统论/物理学**：反馈、临界点、杠杆
- **数学/概率**：不确定性、概率、幂律
- **增长/积累**：复利、惯性、熵增

**选择标准**：哪个领域的模型能揭示**此问题中最容易被忽略的维度**？不要选显而易见的——如果问题明显是经济学问题，经济学视角用户自己也会想到。更有价值的是叠加心理学或系统论视角。

### C. 复杂度 → 选择编排层级

| 复杂度 | 信号 | 策略 |
|--------|------|------|
| 简单 | 单一因素，答案明确 | 1 个实用模型，无需框架 |
| 中等 | 多个因素，需要权衡 | 实用模型 + 领域镜头 |
| 复杂 | 高不确定性，多利益相关方 | 元框架 + 模型 + 镜头 |
| 混沌 | 无先例，危机情况 | OODA 快速循环 → 先稳定局面 |

复杂问题需选择元框架：

| 场景 | 框架 |
|------|------|
| 完整解决方案生命周期 | CGEC（见附录 A） |
| 先发散再收敛 | Diverge-Converge |
| 迭代改进 | PAR（Plan-Act-Reflect） |
| 进入新市场/领域 | Outside-In |
| 挑战假设 | Devil's Advocate |
| 紧急响应 | OODA Loop |
| 以用户为中心的设计 | Double Diamond |
| 全角度审视 | Six Thinking Hats |
| 问题类型不明 | Cynefin（先分类） |
| 目标明确，路径不清 | Working Backwards |
| 反复出现的结构性问题 | Iceberg Model |

## 第 4 步：执行分析

1. 简要说明选择的模型、镜头和理由
2. 复合问题按子问题顺序处理
3. 在每个分析步骤中应用领域镜头，行内标注：*(Economics: 机会成本)*、*(Psychology: 锚定)*
4. 在得出结论前，始终考虑逆向思维："这会怎样失败？"

---

## 第 5 步：不寻常的角度

完成主要分析后，生成 1-2 个不寻常但合理的视角。策略包括：

**学科迁移**：用完全不同学科的框架看同一问题。
- 商业问题 → 生物学镜头（"如果这个公司是一个生态系统..."）
- 技术问题 → 经济学镜头（"这个架构的机会成本是..."）

**时间尺度切换**：
- 用户关注短期 → 给一个 10 年视角
- 用户关注长期 → 问"如果必须一周内决定呢？"

**利益相关者切换**：
- 从竞争对手 / 客户 / 监管者 / 未来的自己的视角重新审视

**反事实**：
- "如果这个约束不存在会怎样？"
- "如果竞争对手先做了这件事，你会怎么应对？"

输出格式：
> **换个角度**：[一句话概括不寻常的视角]
> [2-3 句说明为什么这个角度有价值以及它揭示了什么]

---

## 第 6 步：交叉验证

从三个维度检查分析结果：

### 收敛检验
2 个以上不同学科的模型指向同一结论 → 标记为高置信度。说明为什么多条路径汇聚。

### 矛盾检验
不同模型给出相反建议 → **这是最有价值的发现**。不要回避它。而是：
- 明确说明矛盾点
- 分析矛盾的根源（不同假设？不同时间尺度？不同利益方？）
- 矛盾本身往往揭示了问题的核心张力

### 盲区检验
- 分析中缺失了哪个利益相关方的视角？
- 哪个时间尺度没有覆盖？
- 有没有"房间里的大象"被礼貌地回避了？

---

## 第 7 步：输出与迭代

### 适应性结构

根据问题复杂度调整输出深度：

**简单**：结论 + 关键推理 + 1 个追问
**中等**：诊断 → 分析 → 交叉验证 → 结论 → 不寻常角度
**复杂**：诊断 → 多阶段分析 → 交叉验证 → 矛盾与张力 → 建议 → 不寻常角度 → 追问

### 始终包含
- 分析全程行内标注镜头来源：*(Economics: 机会成本)*、*(Systems: 反馈循环)*
- 结论附带置信度和关键假设
- 至少 1 个追问，引导用户进一步思考

### 分析交付后，提供选项
- 更换镜头获取新视角
- 聚焦某个子问题
- 升级到元框架进行更深入分析
- Devil's Advocate 挑战结论

## 附录 A：自定义框架定义

### CGEC（澄清 → 生成 → 评估 → 沟通）

适用于需要完整解决方案的复杂问题的四阶段框架：

- **Clarify（澄清）**：用 5 Whys / MECE 揭示问题的本质
- **Generate（生成）**：用 ToT / SCAMPER 产生多条解决路径
- **Evaluate（评估）**：用 Inversion / Second-Order 评估风险与长期影响
- **Communicate（沟通）**：用 Golden Circle / Minto 结构化输出

最佳适用：需要从理解到行动到表达的完整生命周期的问题。

---

## 附录 B：反模式清单

避免以下失败模式：

- 选了 5+ 个模型但每个都浅尝辄止 → 宁可 2-3 个用深用透
- 所有领域镜头全选 → 强制只选 2-3 个最有区分度的
- 分析很长但没有明确结论 → 必须给出可执行的建议
- 结论与常识完全一致 → 检查是否遗漏了不寻常的角度
- 回避矛盾，强行统一结论 → 矛盾本身是关键洞察
- 用模型名称堆砌显得专业 → 用户要的是洞察，不是术语展示
